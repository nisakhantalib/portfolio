Technical walkthrough

How Bangkit works, end to end

A student types a question in a browser and gets back a curriculum-grounded answer with a citation. Between those two moments sits a multi-agent AI system, a retrieval pipeline, and a containerised service running in the cloud. Here is every layer, in order.

Next.js frontend Python · FastAPI LangGraph agents RAG retrieval Docker · Azure

01 The mental model

One question, four layers

Before the detail, the shape of the whole thing. The system is not one program — it is four separate pieces that hand work to each other, each running in a different place.

The tutoring-centre analogy. A student walks in with a question. A receptionist doesn't answer it — they work out what kind of help is needed and route the student to the right specialist. One specialist explains concepts, one writes practice quizzes, one grades answers. And before any of them speaks, they walk to a filing cabinet, pull the exact textbook page, and answer from that — telling you which page they used. The receptionist is the supervisor, the specialists are the agents, the filing cabinet is the vector store, and "which page" is the citation.

The request flows left to right; the answer flows back right to left.

The key word is agentic. A plain chatbot is one model answering everything. This system has a coordinator that decomposes a request into a plan and routes each step to a specialised agent — and every answer is grounded in retrieved source material rather than invented. Those two properties are what the rest of this page unpacks.

03 Layer by layer

The frontend: a thin, safe proxy

The Next.js app on Vercel is deliberately not where the intelligence lives. When a student sends a chat message, an API route on the server side (never the browser) forwards it to the agent service. Two design decisions matter here:

  • Server-to-server only. The call to the agent service carries a secret API key in an X-API-Key header. That key lives in Vercel's server environment and never reaches the browser, so a visitor can't read it and rack up your LLM bill.
  • Strangler-fig migration. The original app called the LLM directly. Rather than rip that out, the new path is opt-in: if AI_SERVICE_URL is set, requests route to the agent service; if the service is unreachable, the code silently falls back to the old direct path. Zero downtime, instant rollback.
// web/lib/agentService.js — the proxy call (simplified)
export async function callAgentService({ request, subject, chapter }) {
  const res = await fetch(`${BASE_URL}/v1/agent`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "X-API-Key": API_KEY,        // server-side secret
    },
    body: JSON.stringify({ request, subject, chapter }),
    signal: AbortSignal.timeout(30000), // don't hang forever
  });
  return res.json();
}

The agent service: where the thinking happens

This is the Python system built from scratch — a FastAPI web server wrapping a LangGraph state machine. FastAPI handles HTTP; LangGraph orchestrates the agents. When a request hits POST /v1/agent, it enters a compiled graph as a shared state object that every node can read and write.

The supervisor routes to a worker; each worker returns to "advance", which loops back for multi-step plans or ends.

▸ Deep-dive: what "shared state" actually means

LangGraph passes a single typed dictionary — the state — through every node. A node reads the fields it needs and returns only the fields it changes; LangGraph merges them. This is why the tutor node can see what the supervisor decided without them calling each other directly.

class TutorState(TypedDict, total=False):
    request: str            # the student's question
    intent: str             # tutor | quiz | mark  (set by supervisor)
    plan: list[str]         # ordered steps for multi-part requests
    subject: str            # "sains" | "matematik"
    chapter: str
    retrieved: list[dict]   # RAG results (set by retrieve node)
    answer: str             # final text (set by tutor node)
    quiz: dict              # validated quiz (set by quiz node)

04 The agentic core

The supervisor: decomposition and routing

This node is what makes the system "agentic" rather than a single prompt. It does three jobs on every request:

  • Intent classification. Is this a request to be tutored, to be quizzed, or to have answers marked? A fast, cheap model reads the question and returns a structured verdict.
  • Scope extraction. Which subject and chapter? If the caller already stated it, that wins; otherwise the supervisor infers it.
  • Task decomposition. A compound request like "explain chapter 1, then quiz me" becomes an ordered plan: ["tutor", "quiz"]. The graph executes each step in turn.

Model tiering

The supervisor uses a small, fast model (routing is easy). The tutor and marker use a larger model (reasoning is hard). Right tool, right cost.

Caller scope wins

If the request already specifies subject and chapter, the supervisor never overrides it with a guess. Explicit input beats inference — a real bug fix, see below.

▸ Deep-dive: the decomposition prompt and its safety net

The supervisor asks the LLM for JSON only, then parses it. But LLMs sometimes return malformed output — so there's a fallback: keyword heuristics extract subject and chapter from the raw text. If the model call fails entirely, the system still works, just more crudely. Never a hard failure.

# the LLM is asked for structured intent
"Respond with ONLY: {intents: [...], subject: ..., chapter: ...}"

# but caller-provided scope is authoritative
subject = caller_subject or parsed.get("subject") or heuristic
chapter = caller_chapter or parsed.get("chapter") or heuristic

Retrieval: why answers don't hallucinate

Before any worker answers, a retrieve step runs. This is the RAG (Retrieval-Augmented Generation) pipeline, and it's the reason the tutor cites real chapters instead of making things up. Here's how the curriculum becomes searchable:

Top row runs once when the image is built; bottom row runs on every question.

The tutor node then gets a strict instruction: answer only from this retrieved context, and cite the section. If retrieval returns nothing relevant, the tutor is told to say so rather than invent an answer. That single constraint is the difference between a study tool teachers can trust and a confident liar.

▸ Deep-dive: structured output with self-healing

The quiz and marking agents must return machine-readable JSON matching an exact schema (Pydantic models). Raw LLM output is unreliable, so each structured node validates the response — and if validation fails, it retries once with the validation errors fed back into a repair prompt. Contract enforced at the boundary, not hoped for.

result, error = _structured(llm, prompt, QuizSpec)
# 1. call LLM  2. validate against QuizSpec
# 3. on failure: retry with the error in a repair prompt
# 4. still bad: degrade gracefully, never crash

05 Getting it live

From code to a live URL on Azure

Writing the agent service is one skill; hosting it on the internet so it updates itself is a separate one. This is where most of the real-world difficulty lived. The pipeline has two halves: packaging, and deployment.

Packaging: the container

The service is built into a Docker image — a self-contained box holding the Python code, its dependencies, and the pre-built curriculum corpus. A multi-stage build first runs a Node step to export the curriculum into a searchable file, then assembles a slim Python runtime. The same image runs identically on a laptop, in CI, or in the cloud — that portability is the whole point.

Deployment: automated on every merge

When code is merged to master, GitHub Actions logs into Azure, builds the image, pushes it to a registry, and rolls it out to Azure Container Apps — which scales to zero when idle, so the demo costs nothing between visits.

OIDC federation means no long-lived cloud password is ever stored in GitHub.

▸ Deep-dive: why OIDC instead of a stored key

Instead of saving an Azure password in GitHub, Azure is told to trust tokens GitHub signs — but only when the token says it comes from this exact repository on this exact branch. GitHub mints a fresh signed token per run; Azure verifies the signature and the claim, then issues a short-lived session. Nothing long-lived is stored anywhere. If a token leaked, it's already expired and only ever named one repo.

06 What actually went wrong

Seven production-only failures

Every bug below passed local tests and CI, then failed only in production — because the local and production environments differed in permissions, network, or configuration. This category (the "dev/prod parity gap") is the hardest in real software, and each was diagnosed by reading logs and fixing the root cause.

01

Packaging error

The Python package wouldn't install in CI because setuptools couldn't auto-discover the modules in a flat layout. Fixed with explicit package configuration.

02

Cloud build blocked

The free-tier subscription disabled Azure's cloud build service. Moved the image build into GitHub Actions instead — same image, earlier stage.

03

OIDC claim mismatch

Azure rejected the login because GitHub had moved to immutable subject claims embedding account and repo IDs. Tightened the trust rule to the exact claim.

04

Embedder crashed the container

The embedding library was never added to dependencies; a lazy import passed CI and crashed only at container startup. Added the dep, pre-baked the model, made it degrade gracefully.

05

Empty vector store

An embedding failure during ingestion left the store empty, so every answer lost its grounding. Made the whole build-index step fall back reliably.

06

Caller scope overridden

The supervisor's guess overrode the caller's explicit subject and chapter, so the retrieval filter matched nothing. Made explicit input authoritative over inference.

07

Stale deployment

A green pipeline deployed an old commit because the fix was never merged. Lesson: a successful deploy means "the pipeline ran," not "my latest code is live" — always check the deployed commit SHA.

The through-line: a green pipeline proves the pipeline ran, not that your intended code is live. Reading logs, forming a hypothesis, and fixing the root cause — not the symptom — is the actual skill on display here.

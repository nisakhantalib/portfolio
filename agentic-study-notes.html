01LangGraph 101 — the raw mechanicssource: LangGraph101-v1.ipynb (148 cells)

Everything else builds on four primitives: a state (TypedDict) that flows through the graph, nodes (functions taking state, returning changes), edges (fixed transitions), and routers (functions returning the name of the next node).

# The state — a TypedDict every node reads and updates
class AuthState(TypedDict):
    username: Optional[str]
    password: Optional[str]
    is_authenticated: Optional[bool]
    output: Optional[str]

# A router — reads state, returns the NAME of the next node
def router(state):
    if state['is_authenticated']:
        return "success_node"
    else:
        return "failure_node"

# Assembly — the six calls that build every graph
workflow.add_conditional_edges("ValidateCredential", router,
    {"success_node": "Success", "failure_node": "Failure"})
workflow.set_entry_point("InputNode")
app = workflow.compile()

The notebook's second example (QAState: question → context → answer) is the important one — validate input, fetch context, answer from that context: RAG's shape, minus the actual retrieval.

Maps to bangkit: app/graph/state.py (TutorState is this, richer), app/graph/build.py (same six assembly calls), _dispatch() (same router contract, but routing on an LLM-generated plan instead of if/else). One thing bangkit adds the notebook doesn't teach: nodes take (state, deps) — dependency injection, which is why the tests can swap in fakes.

02Reflection agent — generate → critique → loopsource: Tweet-Reflection-Agent-v1.ipynb (81 cells)

Two LLM roles alternate over a shared message list: a generator produces (a LinkedIn post), a reflector critiques it. The trick that makes the loop work: the critique is returned as a HumanMessage, so the generator experiences it as a user asking for changes.

def generation_node(state: Sequence[BaseMessage]) -> List[BaseMessage]:
    generated_post = generate_chain.invoke({"messages": state})
    return [AIMessage(content=generated_post.content)]

def reflection_node(messages: Sequence[BaseMessage]) -> List[BaseMessage]:
    res = reflect_chain.invoke({"messages": messages})
    return [HumanMessage(content=res.content)]   # critique disguised as the user

# stopping condition: crude — message count
def should_continue(state: List[BaseMessage]):
    if len(state) > 6:
        return END
    return "reflect"

Uses MessageGraph (state = a list of messages) rather than a TypedDict — fine for chat-shaped loops, limiting when you need structured fields.

Maps to bangkit: not implemented as a loop. The family idea — critique the output before trusting it — became verify_node, but bounded to one revision and with a structured verdict instead of free-text critique. The magic-number stopping condition (len > 6) is the weakness to remember; Reflexion fixes it.

03Reflexion — critique drives new retrieval, with citationssource: Reflexion_Agent-v1.ipynb (66 cells)

Reflection's upgrade, in two moves. First, the critique is forced into a schema — and crucially it must produce search queries, so the critique states not just what's wrong but what to go look up:

class Reflection(BaseModel):
    missing: str = Field(description="What information is missing")
    superfluous: str = Field(description="What information is unnecessary")

class AnswerQuestion(BaseModel):
    answer: str
    reflection: Reflection                # structured self-critique
    search_queries: List[str]             # the critique DRIVES retrieval

class ReviseAnswer(AnswerQuestion):
    references: List[str] = Field(description="Citations motivating your updated answer.")

Second, those queries are actually executed, results accumulate in a response_list, and a revisor rewrites the answer with a mandatory references section:

def execute_tools(state: List[BaseMessage]) -> List[BaseMessage]:
    last_ai_message = state[-1]
    for tool_call in last_ai_message.tool_calls:
        search_queries = tool_call["args"].get("search_queries", [])
        for query in search_queries:
            result = tavily_tool.invoke(query)      # live web search per query
        ...append ToolMessage(json.dumps(query_results))...

# stopping condition: counts REAL WORK (tool calls), not raw messages
def event_loop(state: List[BaseMessage]) -> str:
    num_iterations = sum(isinstance(item, ToolMessage) for item in state)
    if num_iterations >= MAX_ITERATIONS:
        return END
    return "execute_tools"
Maps to bangkit: studied, not shipped. verify_node is the same corrective family but retrieval-free (checks against context already retrieved) and single-round. Full Reflexion for bangkit would mean: critique generates a query → re-retrieve from your vector store (not the web) → revise → 2–3 bounded rounds. Natural home: iterative quiz-question refinement.

04ReAct — the model decides every stepsource: ReAct-v1.ipynb (75 cells)

A tight two-node cycle: agent ⇄ tools. The model reasons; if it emits tool_calls, they run and results loop back; the moment it answers without asking for a tool, the loop ends. The LLM itself decides when it's done — no plan, no iteration cap in the core pattern.

# Tools are just decorated functions — the docstring is the model's manual
@tool
def search_tool(query: str):
    """Search the web for information using Tavily API."""
    return search.invoke(query)

model_react = chat_prompt | model.bind_tools(tools)

def should_continue(state: AgentState):
    last_message = state["messages"][-1]
    if not last_message.tool_calls:
        return "end"        # model didn't ask for a tool -> done
    return "continue"       # model asked -> run it, loop back

workflow.add_node("agent", call_model)
workflow.add_node("tools", tool_node)
workflow.add_edge("tools", "agent")        # tools ALWAYS return to agent
workflow.add_conditional_edges("agent", should_continue,
    {"continue": "tools", "end": END})

The notebook's exercises (calculator tool, news summarizer) are the practical takeaway: any function with a good docstring becomes a capability the model can choose to invoke.

Maps to bangkit: deliberately NOT the architecture — bangkit is plan-and-execute (supervisor plans once; predictable cost, testable routing). But the calculator exercise is exactly the mechanism a "verify arithmetic instead of trusting the LLM" feature would use — the scoped, justified place for a ReAct-style tool node inside the existing graph.

07Cheat-sheet distillation — the two loops, drawndistilled from the course cheat sheet (IBM, "Build Self-Improving Agents with LangGraph") — rewritten and redrawn, not reproduced

The reflection loop

The generator and reflector are the same LLM in two roles. The reflections object (response + critique) feeds straight back into generation — no outside information ever enters the loop, which is exactly why reflection plateaus: it can polish, but it can't learn a fact it doesn't have.

Internal-only feedback: nothing new enters after step 1 — polish without grounding.

The Reflexion loop

Same shape, two structural additions: the responder's output is a structured object whose critique names search queries, and an execute-tools step actually runs them — so every revision cycle injects fresh outside evidence, and the revisor must attach citations. This is what turns "improved opinion" into "verifiable claim".

External grounding every cycle: the critique names what to fetch; the revision must cite what came back.

The compressed takeaway (and the line the whole cheat sheet builds to): complexity buys power in a fixed order — reflection is polish, ReAct adds structure, Reflexion adds grounding. bangkit's plan-and-execute sits deliberately outside this spectrum because tutoring steps are enumerable; its verify_node borrows the smallest useful piece of the corrective family without paying any loop's full cost.
